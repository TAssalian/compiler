The computing memory size phase and the code generation phase are designed as two backend passes that run after parsing and semantic analysis. The main reason for this structure is to separate two different jobs. First, the compiler needs to figure out how much memory every part of the program needs and where that memory will be located. After that, it can generate Moon code using those exact sizes and offsets. This keeps the backend organized because the code generation phase does not need to solve layout problems while also producing instructions. It simply uses the memory information that was already computed earlier.

The overall solution is built around the AST, the symbol tables, and two visitors. The AST gives a tree representation of the program, and the symbol tables store the declarations and scope information discovered in the semantic phase. Each SymbolEntry stores useful backend information such as the kind of symbol, its type, its size, and its offset. This means the backend does not need a separate memory model. Instead, it reuses the semantic structures and adds layout information to them. On top of this structure, the ComputeMemSizeVisitor is responsible for assigning sizes and offsets, while the CodeGenVisitor is responsible for producing Moon assembly from that stored layout.

The sequence starts in the backend client. After the parser builds the AST, the compiler creates symbol tables and performs semantic checking. This is important because the memory-size phase depends on already knowing the declared types of variables, the members inside each class, the inheritance relationships between classes, and the inferred types of expressions. Once that information is available, the ComputeMemSizeVisitor walks through the tree and computes memory layout. If there are no blocking semantic errors, the CodeGenVisitor then walks through the same tree and writes the final Moon code.

The purpose of the computing memory size phase is to assign real storage information to each scope. For classes, it computes the size of the full object layout. If a class inherits from other classes, the visitor makes sure the parent class sizes are computed first, then places the inherited memory at the beginning of the child layout. After that, it places the child class’s own data members. This makes the final class size equal to the total memory needed for inherited fields plus newly declared fields.

For functions and the main program block, the memory-size phase computes the stack frame layout. It starts at offset 0 and assigns space to parameters, local variables, and extra hidden entries used by the runtime. In this implementation, two important hidden entries are added: one for the return value and one for the return address. These are treated like normal symbol-table entries, which makes the later code generation phase simpler because it can access them in the same way as any other memory slot.

This phase also decides how large each declared entity is. Integers use 4 bytes, floats use 8 bytes, and addresses use 4 bytes. If a variable has a class type, its size is the size of that class. If a variable is an array, the visitor multiplies the base type size by the array dimensions. A special case is array parameters: instead of reserving the entire array in the callee’s frame, the compiler stores only an address, which matches the implementation of passing arrays by reference-like behavior.

Another important purpose of the memory-size phase is to create storage for values that are not explicitly declared by the programmer. Integer literals, float literals, and intermediate expression results are each given synthetic symbol entries. In practice, this means complex expressions can be evaluated step by step, with each temporary result stored in a known memory location. This design makes the code generator much more uniform because it can treat literals, temporaries, and declared variables in a similar way.

The purpose of the code generation phase is to turn that memory layout into Moon instructions. At the beginning of the program, the CodeGenVisitor emits the entry sequence, initializes the frame pointer register, allocates the main block’s frame, and then branches into the generated program. It also generates labels for functions so each one can be called safely, including overloaded or member functions whose labels need to remain unique.

Inside each function, the code generation phase follows the frame layout created earlier. It stores the return address in the precomputed return-address slot when the function begins, and it restores that address before returning. A return statement copies the expression result into the return-value slot, then jumps back to the caller. Because these memory slots were already created in the previous phase, code generation only has to look up their offsets and emit the proper instructions.

The code generation phase also handles assignments, expressions, control flow, function calls, arrays, and object access. For assignments, it computes the destination address, computes the source value, and copies the value into memory. For expressions, it generates code for the children first, loads their values into registers, performs the operation, and stores the result in the temporary slot that was assigned during memory computation. For if and while statements, it generates labels and branch instructions. For arrays, it computes the correct address using indexes and element size. For object members, it adds the member offset inside the object layout. For function calls, it prepares arguments inside the callee’s frame, jumps to the function label, and restores the frame pointer after the call.

So the implementation can be understood as a simple sequence of backend phases. First, symbol table creation builds the scope structure of the program. Second, semantic checking verifies that the program is meaningful and records expression types. Third, the computing memory size phase assigns sizes and offsets for class layouts, stack frames, variables, literals, temporaries, and hidden runtime values. Fourth, the code generation phase emits Moon code that uses those exact sizes and offsets to run the program correctly. The main strength of this design is that each phase has one clear responsibility, and each one prepares information that makes the next phase easier to implement.
