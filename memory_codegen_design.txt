The computing memory size phase and the code generation phase are both built on top of the AST and the symbol-table information produced earlier in the compiler. The general idea is that the compiler first figures out what every name means and what scope it belongs to, then it computes how much memory each thing needs and where it will live, and only after that does it generate Moon code that uses those exact sizes and offsets. This keeps the backend simple because code generation never has to guess where a variable, parameter, return value, temporary, or object field is stored. That information has already been attached to symbol table entries by the time code generation begins.

The overall structure of the solution is centered around two backend visitors. The first one is the ComputeMemSizeVisitor, and its role is to walk through the AST and assign concrete layout information to each scope. It computes sizes for class data members, function parameters, local variables, literals, temporaries, return values, and return addresses. It also stores offsets inside the corresponding SymbolEntry objects so later phases can access memory uniformly. The second one is the CodeGenVisitor, and its role is to traverse the same tree again and emit Moon assembly code that uses those stored offsets. In other words, the first visitor decides the memory layout, and the second visitor turns the program into executable instructions based on that layout.

This design works well because the SymbolTable and SymbolEntry classes act as the shared storage between phases. A SymbolTable represents one scope such as the global scope, a class scope, or a function scope. A SymbolEntry represents one declared entity such as a class, variable, parameter, or function, and also stores the extra backend information like size and offset. Because these objects are already attached to AST nodes during semantic analysis, the memory visitor and the code generation visitor can reuse the same structure instead of building separate backend-only representations.

The process starts in the backend client after parsing and semantic analysis succeed. The compiler first builds symbol tables with the SymTabCreationVisitor, then checks semantic rules with the SemanticCheckingVisitor. Only after these earlier phases succeed enough for layout to make sense does it run the ComputeMemSizeVisitor. This ordering matters because memory layout depends on semantic facts such as the declared type of each variable, the members of each class, the inheritance relationships between classes, and the inferred type of each expression. Once those sizes and offsets are available, the compiler runs the CodeGenVisitor and writes the final Moon program to the output file.

The memory computation phase itself is organized by scope. When the visitor enters the global program node, it records the global table and then visits the classes, free functions, and main program block. When it enters a class, it starts a fresh offset counter for that class layout. If the class inherits from parent classes, it makes sure each parent class has already had its size computed, then places each inherited class block into the child object layout one after another. The visitor records where each parent subobject begins through inherited_class_offsets, and then continues by laying out the child class' own data members after the inherited memory. This means the final class size is the total size of all inherited data plus the size of the child’s declared members.

When the memory visitor enters a function definition or the main program block, it also starts a fresh offset counter for that stack frame. It visits formal parameters first, then local declarations inside the body, and finally reserves hidden storage used by the runtime calling convention. In this implementation, that hidden storage includes a synthetic entry for the return value and another synthetic entry for the return address. These entries are created like normal symbol table entries, which means the frame layout stays consistent and the code generator can access them exactly like any other slot in memory.

Variable sizing is handled in a direct way. Integers take 4 bytes, floats take 8 bytes, and addresses take 4 bytes. If a variable has a class type, its size is the size of that class table, so object variables physically reserve enough memory to contain the whole object. If a variable is an array, the visitor multiplies the base type size by each known dimension. There is one important special case for array parameters: instead of copying the whole array into the function frame, the implementation stores only an address for the parameter, so array parameters use address size rather than full array size. This matches the later code generation logic, which passes arrays by address.

Another important part of the memory phase is the creation of synthetic storage for values that do not come from explicit declarations. Integer and float literals are each given their own synthetic symbol entries, and expression nodes such as unary operators, arithmetic operators, and relational operators also receive temporary entries. The reason for this is simple: the code generator evaluates expressions by storing intermediate results in memory slots, not only in registers. By allocating those temporary slots in advance, expression evaluation becomes uniform. Every meaningful value in the backend can be treated as something that either already has an address or can be given one.

The code generation phase begins after all of that layout information already exists. The CodeGenVisitor keeps two instruction streams, one for executable instructions and one for reserved data such as the output buffer. It also keeps track of the current scope so it knows which symbol table to use when resolving offsets. At the start of the program, it emits the Moon entry sequence, initializes the frame pointer register r14 to topaddr, allocates the main block’s frame by subtracting its computed size, and jumps to the generated label for main. After that it emits code for functions and for the main block, then finishes with hlt.

Function generation follows the frame layout created earlier. Each function gets a unique label based on its name, parameter types, and owning class when applicable, which avoids collisions between overloads and member functions. On entry, the function stores the return address register into the precomputed __ret_addr slot. On exit, it reloads that saved return address and jumps back to the caller. Return statements copy the computed expression value into the function’s __ret_val slot before branching back. Because the return slot and return-address slot were both inserted during memory computation, the code generator only needs to look them up and emit the proper load and store instructions.

Assignments, variable accesses, member accesses, array accesses, and function calls are all handled through address computation. The visitor resolves a chain like a.b[2].c(x) from left to right. The first part of the chain may refer to a local variable, parameter, object variable, member function, or data member. Later parts treat the previously computed address as the base and add field offsets or apply indexing logic. For object members, field access works because class data members were already laid out with precise offsets inside the object. For arrays, the visitor multiplies indexes by the proper stride using the remaining dimensions and element size. For array parameters, it first loads the stored address and then indexes from that address rather than from a local frame slot.

Function calls use the callee’s symbol table and frame size directly. For each argument, the code generator looks at the matching parameter entry in the callee’s table and copies the argument into the caller-prepared part of the callee frame. Ordinary parameters are copied by value, while array parameters pass an address. If the call is a member function call on an object, the object address is also stored so the callee can access its data members. Then the code generator adjusts r14 downward by the callee frame size, performs the jump-and-link instruction, and restores r14 after the call returns. This is why computing each function table’s total size beforehand is necessary: the call sequence depends on that exact frame size.

Expression code generation is also sequential and consistent with the memory design. Literal nodes first place their values into their synthetic memory slots. Unary and binary expression nodes recursively generate code for their children, load the child values from memory into registers, perform the Moon operation, and store the result into the temporary slot assigned to the current node. This means even a large complex expression is reduced into a series of small steps where each intermediate result has its own known location. The write statement then simply loads the final expression value, calls the Moon library routines to convert it to text, and prints it.

So, in terms of phases, the backend implementation can be described as a clean sequence. First, symbol table creation establishes scopes and attaches entries to declarations. Second, semantic checking validates types and resolves the meaning of expressions and references. Third, memory-size computation converts those validated declarations and inferred expression types into concrete storage sizes and offsets for classes, frames, literals, temporaries, and hidden runtime slots. Fourth, code generation uses that stored layout information to emit Moon instructions for control flow, expression evaluation, data access, function calls, returns, and output. The important point is that each phase only does one main job and hands structured information to the next one, which is why the whole backend remains manageable even when handling classes, arrays, functions, and nested expressions.
